## What This Does

Before a student pays ₹100 to apply for a job, this system checks their
skill profile against the job's requirements. If the match score is too low,
the payment is blocked and the student receives a clear explanation of why —
along with advice on which skills to build before trying again.

```
Student Profile → Matching Model → Match Score → Guardrail Decision → Allow / Warn
```

## Quick Start

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Generate data (already included, but can be re-run)

```bash
cd data
python generate_data.py
```

### 3. Run the notebook

```bash
cd notebooks
jupyter notebook guardrail.ipynb
```

Or as a plain script:

```bash
python guardrail.py
```

### 4. Start the API

```bash
cd api
uvicorn app:app --reload
```

Then visit: http://localhost:8000/docs

## Guardrail Thresholds

| Score Range | Status         | Payment |
|-------------|----------------|---------|
| 85 – 100    | Excellent Match | ✅ Allowed |
| 70 – 84     | Good Match      | ✅ Allowed |
| 50 – 69     | Moderate Match  | ✅ Allowed |
| 25 – 49     | Low Match       | 🚫 Blocked |
| 0 – 24      | Very Low Match  | 🚫 Blocked |

## API Usage

### Check before payment

```bash
curl -X POST http://localhost:8000/check-match \
  -H "Content-Type: application/json" \
  -d '{"student_id": 15, "job_id": 3}'
```

Response:

```json
{
  "student_id": 15,
  "job_id": 3,
  "match_score": 42.0,
  "status": "LOW_MATCH",
  "status_label": "⚠️  Low Match",
  "message": "Low chance of shortlisting...",
  "matched_skills": ["python"],
  "missing_skills": ["sql", "machine learning", "statistics"],
  "allow_payment": false,
  "application_fee": 100,
  "advice": "You're missing sql, machine learning, statistics..."
}
```

### Other endpoints

```
GET /student/{id}    → student profile
GET /job/{id}        → job details
GET /metrics         → guardrail evaluation metrics
GET /students        → list students
GET /jobs            → list jobs
GET /health          → liveness check
```

## Evaluation Metrics

Run the notebook to get live numbers. Typical results on the 4000-pair dataset:

| Metric              | Value  |
|---------------------|--------|
| Precision           | 1.000  |
| Recall              | 1.000  |
| F1 Score            | 1.000  |
| False Positive Rate | 0.000  |

These are deterministic since the guardrail threshold (≥50) maps exactly to
the ground truth label (score ≥ 50 = good match). The more interesting
comparison is against the **baseline** (always block everyone): our model
improves precision by ~13pp while maintaining full recall.

## Notes

- Matching is skill-overlap based (coverage ratio). This is intentionally
  simple and fully explainable — every score comes with a breakdown.
- The guardrail threshold (50%) is a business rule, not a model parameter.
  It can be adjusted in `models/guardrail.py` under `THRESHOLDS`.
- Payment failure handling (network errors, gateway timeouts) is handled
  by the backend team. This module only decides *whether* to show the
  payment button.
